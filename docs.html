Operate LiteHCP Targeter

Everything from first upload to production operation. The app is one PHP file; this page is the rest of the manual.

DOC · 01Deploy on shared hosting (cPanel / Namecheap)

Requirements: PHP 8.0+ with PDO + SQLite (default on nearly every host), and write permission in the app directory.

# 1. upload the single file
public_html/litehcp.php

# 2. cPanel → MultiPHP Manager → set directory to PHP 8+

# 3. permissions
litehcp.php        0644
app folder         0755
litehcp_uploads/   0750   # created automatically; 0755 if your host requires it

# 4. enforce HTTPS (hosting panel or redirect rule)

# 5. visit /litehcp.php → register → first user becomes admin

On first load the app auto-creates its SQLite schema at ./litehcp.db and the litehcp_uploads/ directory (with web access denied via .htaccess).

See it working immediately: Dashboard → Load sample imports the embedded demo dataset — 200 realistic HCP profiles with believable data gaps, 8 enrichment rules, and 5 segments. Same data as the browser demo.

DOC · 02Run locally

$ git clone https://github.com/tanzir71/litehcp_targeter
$ cd litehcp_targeter
$ php -S localhost:8000
# → open http://localhost:8000/litehcp.php

The built-in PHP server is fine for evaluation and development. For production, use any real web server with HTTPS.

DOC · 03Configuration

All knobs live in the $CONFIG block at the top of litehcp.php — priority weights, confidence penalties, imputation defaults, export field allowlists, rate limits, upload limits. A few common settings can be overridden without editing the file via .env next to it:

# .env (never commit this file)
CRON_TOKEN="change-this-long-random-token"
ACCENT_COLOR="#2540FF"
SESSION_TIMEOUT_SECONDS="1800"
UPLOAD_MAX_BYTES="5242880"
Config keyDefaultWhat it does
priorityWeights0.30 / 0.40 / 0.30Consent / recency / engagement weights for the base priority score.
recencyWindowDays90Activity older than this contributes zero recency.
confidenceStrictThreshold70Minimum confidence for strict-mode segments (also editable in Settings).
confidence*Penalty10 / 3 / 6Confidence deductions for missing required, missing optional, and imputed fields.
importCommitEvery200Rows per import chunk/transaction.
defaultExportFieldsMinimal6 fields, no PIIWhat a standard export contains.
rateLimitlogin / register / exportWindow, attempt, and block settings per action.

DOC · 04CSV import

Import is a three-step flow: upload → map → run.

  • Upload — CSV only, validated by extension, MIME type, and size (UPLOAD_MAX_BYTES). Filenames are sanitized and files stored in litehcp_uploads/.
  • Map — match your CSV columns to internal fields. Internal fields: hcp_id, name, email, specialty, region, organization, role, consent_email, consent_web, last_activity_ts, imports_count. A mapping-confidence figure feeds the profile confidence score.
  • Run — processes in chunks (200 rows by default) with a progress bar and a "run until done" auto-reload mode. Profiles upsert by hcp_id, so re-imports are safe.

DOC · 05Imputation

Each internal field gets an imputation strategy applied to blank values during import:

StrategyBehavior
fixedSubstitute a constant (e.g. region → "Unknown").
modeUse the most frequent value in the file for that column.
meanUse the column mean (numeric fields).
leave_null_flagKeep it null, but flag the profile as incomplete.

Imputed values are stored in the profile's metadata and shown with a dotted underline throughout the UI — imputation is never silent.

DOC · 06Scoring

Priority score (0–100)

priority = 100 × ( 0.30 × consent01      # (consent_email + consent_web) / 2
                 + 0.40 × recency01      # 1 − age/recencyWindow, floored at 0
                 + 0.30 × engagement01 ) # imports_count / 10, capped at 1

Rules may then adjust it (add_priority_delta) or set it outright (set_priority_score). The final value is clamped to 0–100.

Confidence score (0–100)

Starts at 100 and is reduced by missing required fields, missing optional fields, imputed fields, and incomplete column mapping. Strict-mode segments and exports can require a minimum confidence.

DOC · 07Rules reference

Rules are JSON, evaluated against every profile in priority order (highest first). A rule is a condition tree plus actions:

{
  "match": "AND",                     // or "OR"; nested "all"/"any" arrays also supported
  "conditions": [
    { "field": "specialty", "op": "=",  "value": "oncology" },
    { "field": "consent_email", "op": "=", "value": 1 }
  ],
  "continue_on_match": true           // false = stop evaluating further rules
}
OperatorsActions
= · != · > · >= · < · <= · in · contains · regex set_persona · set_priority_score · add_priority_delta · set_compliance_flag · add_tags

The special field tags matches against tags added by earlier (higher-priority) rules, so rules can chain. Every matched rule is recorded in the profile's metadata trace — segments can target "profiles where rule X fired".

Test before saving: the rule editor includes a tester that evaluates a draft rule against up to 50 live profiles and shows exactly which match and why.

DOC · 08Segments

A segment is either a safe SQL-like filter or a set of rule IDs (profiles where those rules fired). Filters are strictly validated: field names must be on the allowlist, and semicolons, comments, double quotes, and DDL/DML keywords are rejected.

-- examples (all from the embedded sample)
specialty = 'oncology' AND confidence_score >= 70
consent_email = 1 AND region = 'West'
imports_count >= 5
persona = 'Do Not Contact'

Two run modes: strict additionally requires confidence_score ≥ threshold (default 70); inclusive returns all matches. Preview member counts and rows before exporting.

DOC · 09Campaign simulator

Pick a segment, set assumptions — contact rate, response rate, cost per contact, value per response — and get projected contacts, responses, cost, value, and ROI. Every run is stored in the audit log with its assumptions, so the numbers behind a plan stay traceable.

DOC · 10Exports & privacy

  • Default export fields: hcp_id, specialty, region, persona, priority_score, confidence_score — no email, no name.
  • PII export (email/name) requires an admin to enable export_pii in Settings, and each PII export is individually audited.
  • Field selection is enforced against allowedExportFields server-side.
  • Exports are rate-limited.

DOC · 11Cron recompute

For large datasets, recompute base scores and re-apply rules on a schedule instead of synchronously:

# cPanel → Cron Jobs (CLI form, preferred)
php /home/USER/public_html/litehcp.php action=cron_recompute token=YOUR_CRON_TOKEN

# or via HTTP
https://your-site.com/litehcp.php?action=cron_recompute&token=YOUR_CRON_TOKEN

Set the token via .env (CRON_TOKEN) and rotate it if it leaks. Processing is batched (200 profiles per transaction).

DOC · 12Security model

  • Auth — password hashing via password_hash, session regeneration on login, idle timeout, first user becomes admin, registration closed by default afterward.
  • CSRF — token required on every POST.
  • Headers — CSP (nonce-based scripts, no third-party origins beyond fonts), X-Frame-Options DENY, nosniff, referrer policy, permissions policy.
  • SQL — prepared statements everywhere; segment filters validated against a field allowlist with keyword denylist.
  • Uploads — CSV-only validation by extension/MIME/size, sanitized filenames, uploads directory denied to the web.
  • Rate limiting — login, registration, and export endpoints.
  • Errors — logged server-side; no stack traces to the browser.

Hardening checklist for production: enforce HTTPS, use a strong admin password, consider an .htaccess IP allowlist or basic auth in front, and make sure litehcp.db is not world-readable. Full details in SECURITY.md.

DOC · 13Backups

The entire state is two artifacts next to the app file:

litehcp.db          # all data — back up daily/weekly
litehcp_uploads/    # original CSVs + server logs — optional

Restoring is copying the files back. There is no external state.
